Module

Description

The java-sdk module provides a Java API for defining continuous deployment pipelines with container-based execution. It offers a fluent builder API that enables developers to construct pipeline definitions programmatically, which are then serialized to protobuf format for language-agnostic execution by the controller.

This module is the primary interface for pipeline authors. Rather than writing complex YAML configurations, developers use type-safe Java code to define their deployment pipelines. The module includes an annotation processor that generates the necessary boilerplate code for pipeline serialization and execution.

Why Java SDK?

Traditional CI/CD systems often rely on YAML configuration files that can become unwieldy for complex pipelines. The Java SDK addresses this by providing:

  • Type Safety: Compile-time checking of pipeline definitions prevents many common errors
  • IDE Support: Auto-completion, refactoring, and navigation work seamlessly
  • Code Reuse: Common pipeline patterns can be extracted into reusable functions and classes
  • Testing: Pipeline logic can be unit tested using the java-test module
  • Dynamic Generation: Pipelines can be generated at runtime based on context and parameters

Responsibilities

  • Provide fluent API for pipeline definition
  • Define work items and their dependencies
  • Support multiple work definition types (containerized, custom, compound, dynamic)
  • Serialize pipeline definitions to protobuf format
  • Annotation processor generates pipeline definition runner code
  • Manage work context and parameter passing between work items

Dependencies

  • protocol (protobuf definitions)

Packages

yeetcd.sdk

Core SDK classes for pipeline definition. Provides the main API for constructing pipelines, defining work items, and configuring work dependencies. Contains the essential classes: Pipeline, Work, WorkDefinition, and the annotation processor.

yeetcd.sdk.condition

Condition classes for controlling work execution flow. Conditions determine whether a work item should execute based on work context values or previous work execution status. Supports logical composition (AND, OR, NOT) of conditions.

Contracts

Unique Pipeline Names: All pipelines must have a unique name within a project scope to prevent conflicts in the native work registry.
Immutable Definitions: Pipeline definitions are immutable once created via the builder pattern, ensuring thread safety and preventing accidental modifications.
Deterministic Work IDs: Work IDs are deterministically generated based on work content and context, enabling consistent identification across executions.
Native Work Registration: Native work definitions are automatically registered in a static map keyed by pipeline name for runtime execution.
Deterministic Serialization: All protobuf serialization is deterministic for equivalent pipeline definitions, ensuring consistent output.

Invariants

Valid Pipeline Names: Pipeline names are never null or empty.
Unique Work IDs: Work IDs are unique within a pipeline context.
Valid Execution Identifiers: All work definitions have valid execution identifiers.
Non-null Context: Work context maps are never null (empty context is represented as empty map).
Acyclic Dependencies: Previous work references form a directed acyclic graph (no circular dependencies).

Usage Example

Here's a simple example of defining a pipeline using the Java SDK:

@PipelineGenerator
public static Pipeline myPipeline() {
    // Define a containerized work item
    Work build = Work.builder("Build application",
        ContainerisedWorkDefinition.builder()
            .image("maven:3.9")
            .command(List.of("mvn", "clean", "package"))
            .build())
        .build();
    
    // Define a work item that depends on the build
    Work test = Work.builder("Run tests",
        ContainerisedWorkDefinition.builder()
            .image("maven:3.9")
            .command(List.of("mvn", "test"))
            .build())
        .previousWork(PreviousWork.builder(build).build())
        .build();
    
    // Create the pipeline
    return Pipeline.builder("ci-pipeline")
        .finalWork(test)
        .build();
}
