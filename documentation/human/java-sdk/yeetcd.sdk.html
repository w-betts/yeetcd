What is yeetcd.sdk?

The yeetcd.sdk package provides the core API for defining pipelines in the yeetcd continuous deployment system. It offers a fluent, type-safe builder pattern for constructing complex deployment pipelines with support for containerized execution, custom code execution, and dynamic work generation.

This package serves as the primary interface between developers and the yeetcd platform. Instead of writing YAML or JSON configuration files, developers use the Java SDK to programmatically define their deployment pipelines using intuitive builder patterns. The SDK handles all the complexity of serializing these definitions to Protocol Buffers for transmission to the controller.

Why Does This Package Exist?

Traditional deployment systems often rely on static configuration files that can become unwieldy as complexity grows. The yeetcd.sdk package addresses this by providing:

  • Type Safety: Compile-time checking of pipeline definitions prevents runtime errors
  • Code Reusability: Pipeline components can be shared, versioned, and tested like any other code
  • IDE Support: Full autocomplete, refactoring, and navigation capabilities
  • Testability: Pipelines can be unit tested without requiring actual container execution
  • Flexibility: Dynamic work generation allows runtime decision-making about what work to execute

Core Classes

Pipeline

The root container for a deployment pipeline. Defines the pipeline name, parameters, and the final work item that represents the complete deployment process. Uses an immutable builder pattern for construction.

Work

Represents a single unit of work within a pipeline. Each work item has a description, a work definition (what to execute), optional dependencies on previous work, and optional execution conditions.

WorkDefinition (Interface)

The abstraction for different types of executable work. Implementations include:

  • ContainerisedWorkDefinition: Execute commands in existing container images
  • CustomWorkDefinition: Execute user-defined Java code in a built container
  • CompoundWorkDefinition: Group multiple work items as a single unit
  • DynamicWorkGeneratingWorkDefinition: Generate work at runtime based on parameters

PreviousWork

Defines dependencies between work items. Allows mounting output files from previous work and capturing stdout as environment variables for dependent work.

WorkContext & Parameters

Key-value configuration passed to work items as environment variables. WorkContext applies to individual work items, while Parameters define pipeline-level inputs with validation.

PipelineGenerator (Annotation)

Marks static methods that return Pipeline objects. The annotation processor generates runtime code to discover and execute these methods during the build phase.

Contracts

All builder methods return the builder instance for method chaining
Immutable objects are created via builders only
All public APIs accept non-null values (nulls cause IllegalArgumentException)
Work IDs are deterministically generated from content hash
Native work definitions are automatically registered when pipeline is built

Invariants

No null values accepted by public APIs
All collections are immutable after construction
Work dependencies form a directed acyclic graph
Pipeline names are unique within the static registry

Responsibilities

  • Define Pipeline, Work, and WorkDefinition interfaces
  • Provide builder patterns for pipeline construction
  • Implement work definition types (Containerised, Custom, Compound, Dynamic)
  • Manage work context and parameter passing
  • Support work dependencies via PreviousWork
  • Serialize pipeline definitions to protobuf format
