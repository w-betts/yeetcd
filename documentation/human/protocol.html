Module

Description

The protocol module defines the protobuf schema for language-agnostic pipeline serialization. It serves as the foundation for cross-language communication between the Java SDK (which generates pipeline definitions) and the controller (which executes them).

By using Protocol Buffers as the wire format, yeetcd enables:

  • Language independence: Pipeline definitions can be generated by any language that supports protobuf
  • Backward compatibility: Schema evolution is managed through protobuf's versioning capabilities
  • Efficient serialization: Compact binary format for transmission and storage
  • Type safety: Generated code ensures compile-time correctness

Responsibilities

  • Define protobuf schema for pipeline serialization
  • Enable language-agnostic pipeline representation
  • Support evolution of pipeline format via protobuf versioning

Contracts

Backward Compatibility: Protobuf definitions maintain backward compatibility. New fields can be added without breaking existing code.
Version Consistency: All SDK modules use the same protobuf version to ensure compatibility between generated and consumed messages.

Invariants

Source of Truth: The protobuf schema is the source of truth for the pipeline wire format. All language implementations must conform to this schema.

Key Concepts

Pipeline Message

The root message that represents a complete pipeline definition. It contains:

  • name: Unique identifier for the pipeline
  • parameters: Input parameters that can be passed at execution time
  • workContext: Key-value pairs passed as environment variables to all work items
  • finalWork: The work items that constitute the pipeline execution

Work Message

Represents a single unit of execution within a pipeline. Each work item has:

  • id: Unique identifier generated from content hash
  • description: Human-readable description of the work
  • definition: The actual work to execute (container, custom, compound, or dynamic)
  • dependencies: Previous work items that must complete before this work can run
  • condition: Optional condition that determines if work should execute

Work Definition Types

The protocol supports four types of work definitions:

  • ContainerisedWork: Execute a command in an existing container image
  • CustomWork: Execute user-defined Java code identified by executionId
  • CompoundWork: Group multiple work items as a single unit
  • DynamicWork: Generate work at runtime based on parameters and context
