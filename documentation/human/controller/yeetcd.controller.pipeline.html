What is yeetcd.controller.pipeline?

The yeetcd.controller.pipeline package is the heart of the yeetcd execution engine. It manages the runtime representation of pipelines, orchestrates work execution with proper dependency resolution, tracks work results, and evaluates conditions to determine work eligibility.

This package bridges the gap between the static pipeline definitions created by the Java SDK and the dynamic execution environment. It deserializes protobuf definitions into executable runtime objects, manages the lifecycle of pipeline execution, and coordinates with the execution engine to run work in containers.

Why Does This Package Exist?

Pipeline execution requires sophisticated orchestration capabilities:

  • Dependency Resolution: Work items must execute in the correct order based on their dependencies
  • Condition Evaluation: Runtime decisions about whether work should execute
  • Result Tracking: Capturing and propagating work results to dependent work items
  • Dynamic Generation: Handling work items that generate additional work at runtime
  • Failure Handling: Managing pipeline state when work fails and determining which dependent work should be skipped
  • Event Emission: Providing visibility into pipeline execution through lifecycle events

Core Classes

PipelineController

The main entry point for pipeline execution. Manages the overall execution flow, coordinates work scheduling, handles dynamic work generation, and emits lifecycle events. Ensures work dependencies are resolved before execution and conditions are evaluated.

Pipeline (Runtime)

The runtime representation of a pipeline definition. Contains the pipeline name, the final work item (entry point), and tracks the overall pipeline status. Unlike the SDK's Pipeline class, this version is mutable during execution to track state changes.

Work (Runtime)

Runtime representation of a work item. Contains a unique ID, the work definition (what to execute), optional condition for execution eligibility, and methods for execution via the ExecutionEngine.

WorkResultTracker

Maintains a registry of work results to ensure idempotent execution. Prevents the same work from being executed multiple times within a pipeline run and provides results to dependent work items.

WorkDefinitions

Factory class for deserializing work definitions from protobuf. Maps protobuf work definition types to their corresponding runtime implementations (Containerised, Custom, Compound, Dynamic).

PipelineResult & WorkResult

Result objects capturing the outcome of pipeline and work execution. Include status (SUCCESS, FAILURE, etc.), exit codes, and output summaries. Used for reporting and downstream decision-making.

Conditions (Runtime)

Evaluates condition objects from the SDK at runtime. Receives the current work context and previous work results to determine if a work item should execute.

Contracts

Work is executed at most once per pipeline run
Work dependencies are resolved before execution
Conditions are evaluated before work execution
Failed work causes dependent work to skip (unless overridden by condition)
Dynamic work generation produces valid Work protobuf on stdout
Work results are tracked and accessible to dependent work

Invariants

WorkResultTracker ensures idempotent work execution
Work IDs are unique within a pipeline
Pipeline status reflects aggregate of final work status
Previous work references are immutable after construction

Responsibilities

  • Manage runtime Pipeline objects from protobuf
  • Execute Work items with dependency resolution
  • Track work results via WorkResultTracker
  • Deserialize work definitions from protobuf
  • Evaluate conditions to determine work execution
  • Handle dynamic work generation at runtime
  • Emit pipeline and work lifecycle events
