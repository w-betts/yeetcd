What is yeetcd.controller.execution?

The yeetcd.controller.execution package provides the execution engine abstraction for running containerized jobs. It defines the ExecutionEngine interface and provides implementations for both Docker and Kubernetes, enabling yeetcd to execute work items in isolated container environments.

This package handles the complete lifecycle of container execution: building container images from source artifacts, executing jobs with proper input/output handling, streaming logs, and extracting results. The abstraction allows the controller to switch between Docker and Kubernetes without changing the orchestration logic.

Why Does This Package Exist?

Container execution requires sophisticated infrastructure management:

  • Image Building: Creating container images with user code and dependencies
  • Job Execution: Running containers with proper resource isolation
  • Input/Output Handling: Mounting files and extracting results
  • Log Streaming: Capturing stdout/stderr in real-time
  • Platform Abstraction: Supporting both Docker and Kubernetes
  • Resource Cleanup: Ensuring containers and volumes are cleaned up

Core Classes

ExecutionEngine Interface

Defines the contract for execution engines. Provides methods for building container images and executing jobs. Implementations must be thread-safe and handle async operations via CompletableFuture.

AbstractExecutionEngine

Base class providing common functionality for image building and job execution. Handles shared concerns like image tagging, log streaming, and output extraction. Subclasses implement platform-specific details.

DockerExecutionEngine

Docker-based implementation using the docker-java client. Builds images using Jib and executes jobs as Docker containers. Handles volume mounting, log streaming, and container cleanup.

KubernetesExecutionEngine

Kubernetes-based implementation using the kubernetes-client. Builds images and pushes to a registry, then executes jobs as Kubernetes Jobs. Manages PVCs for persistent storage and handles pod lifecycle.

ExecutionEngines Factory

Factory class for creating execution engine instances based on configuration. Selects Docker or Kubernetes implementation based on the configured execution engine type.

Contracts

ExecutionEngine implementations are thread-safe
Image building produces runnable images with all artifacts
Job execution returns exit code and output directories
Failed jobs (non-zero exit code) still return output directories if available
Log streaming captures both stdout and stderr
Image tags are unique based on source SHA

Invariants

All async operations use CompletableFuture for composition
Thread pools are properly named for monitoring
Container resources are cleaned up after job completion
Image building is deterministic for same inputs

Responsibilities

  • Define ExecutionEngine interface for job execution
  • Provide Docker-based execution engine implementation
  • Provide Kubernetes-based execution engine implementation
  • Build container images from artifact definitions
  • Execute jobs in containers with proper input/output handling
  • Stream container logs to configured output streams
  • Extract output directories from completed jobs
  • Manage persistent volume claims for Kubernetes
